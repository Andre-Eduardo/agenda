# Internationalization (i18n)

## Stack

- **i18next** 24.x — core i18n library
- **react-i18next** 15.x — React bindings
- **`useTranslation`** (from UI library hooks) — app-wide hook (wraps react-i18next)

## Supported Locales

| Locale | Language |
|--------|---------|
| `pt-BR` | Brazilian Portuguese (default) |
| `en-US` | English |
| `es-ES` | Spanish |

The active locale is determined by `navigator.language`. If the browser language is not in the supported list, it falls back to `pt-BR`.

---

## i18next Instance Setup

A single i18next instance is created in `src/translations/i18n.ts` and shared across the app and the UI library:

```ts
import {createInstance} from 'i18next';

export const defaultLocale = {
  code: useAppStore.getState().language,        // from persisted Zustand store
  timeZone: Intl.DateTimeFormat().resolvedOptions().timeZone,
};

export const i18nInstance = createInstance({
  ns: ['ui', 'app', 'common', 'country'],       // namespaces
  lng: defaultLocale.code,
  fallbackLng: defaultLocale.code,
  load: 'currentOnly',                           // don't load 'pt' when 'pt-BR' is set
  interpolation: {escapeValue: false},
  supportedLngs: ['pt-BR', 'en-US', 'es-ES'],
  resources: {
    'pt-BR': {...uiPtBr, ...appPtBr},           // merge UI library + app translations
    'en-US': {...uiEnUs, ...appEnUs},
    'es-ES': {...uiEsEs, ...appEsEs},
  },
});

// Initialized in App.tsx (fire-and-forget)
i18nInstance.init();
```

The instance is passed to the UI library's theme provider to share locale context.

---

## Namespaces

| Namespace | Content |
|-----------|---------|
| `ui` | UI library translations (component labels, generic errors, date formats) |
| `app` | All app-specific feature translations |
| `common` | Shared cross-feature strings (e.g., `permission.denied`, `action.save`) |
| `country` | Country names |

---

## Translation File Structure

```
src/translations/
├── i18n.ts             # i18next instance config
├── pt-BR/
│   ├── index.ts        # re-exports all namespaces for this locale
│   ├── app.ts          # 'app' namespace
│   ├── common.ts       # 'common' namespace
│   └── country.ts      # 'country' namespace
├── en-US/
│   └── ...             # same structure
└── es-ES/
    └── ...             # same structure
```

## Using Translations in Components

### `useTranslation`

```ts
import {useTranslation} from '@ui-lib/hooks/useTranslation';

const {t} = useTranslation('app');
const {t: tCommon} = useTranslation('common');

t('module.featureA.page.list.title')
tCommon('permission.denied')
tCommon('action.save')
```

### Translation Key Constants (`translations.ts`)

Each component and page declares its key prefix in a co-located `translations.ts` to avoid repeating the full path:

```ts
// modules/feature-a/pages/ListPage/translations.ts
export const translationKey = 'app.module.featureA.page.list' as const;
```

Usage in the component:
```ts
const {t} = useTranslation(translationKey);

t('title')                     // → 'app.module.featureA.page.list.title'
t('table.column.name')         // → 'app.module.featureA.page.list.table.column.name'
t('action.create')
t('empty.title')
```

For validation error messages, use a sub-namespace pattern:
```ts
const {t} = useTranslation(`${translationKey}.validation`);
t('name.error.required')
```

## Rich Text with React Elements (`Trans`)

When a translation string needs to embed a React component, use `Trans`:

```tsx
import {Trans} from 'react-i18next';

<Trans
  t={t}
  i18nKey="action.delete.modal.description"
  values={{name: item.name}}
  components={{bold: <Text as="strong" size="sm" weight="bold" />}}
/>
```

Translation string: `"Are you sure you want to delete <bold>{{name}}</bold>?"`

## Language Switching

The active language is stored in Zustand and persisted to localStorage. When the user changes language, the UI library's provider calls `i18nInstance.changeLanguage(language)`:

```tsx
// App.tsx — the UI library handles the actual i18n switch
<ThemeProvider onLanguageChange={setLanguage} ...>
```

---

## Locale for Date and Number Formatting

The `defaultLocale` object (code + timezone) is passed to the UI library provider:

```tsx
<ThemeProvider locale={defaultLocale} i18n={i18nInstance}>
```

This controls date formatting, number formatting, and calendar locale across all UI library components automatically.

## Translation Key Naming Convention

Keys follow a dot-notation hierarchy mirroring the UI structure:

```
{namespace}.{area}.{module}.{sub-area}.{element}.{property}
```

**Examples:**
```
app.module.featureA.page.list.title
app.module.featureA.page.list.table.column.name
app.module.featureA.page.list.action.create
app.module.featureA.page.list.validation.name.error.required
app.module.featureA.page.list.empty.title
app.module.featureA.component.itemForm.form.field.name.label
common.permission.denied
common.action.save
common.action.cancel
```
